# 基于RAG和大语言模型的AI文献助手 Spec

## Why
大学生在学术研究过程中需要快速了解某个研究领域的核心观点，但传统文献检索方式效率低下，需要阅读大量文献才能获取关键信息。通过构建基于RAG技术的AI文献助手，可以帮助大学生快速获取相关文献的核心观点总结，提升研究效率。

## What Changes
- 构建基于LangChain的RAG知识库系统，从Excel数据构建向量数据库
- 实现语义检索功能，根据用户输入的主题检索相关文献
- 集成阿里云通义千问大模型，实现文献核心观点的智能总结
- 开发Web界面，提供用户友好的交互体验
- 实现结构化输出，清晰展示文献列表和核心观点总结

## Impact
- 受影响的能力：文献检索、知识管理、智能总结、Web交互
- 受影响的代码：新建项目，无现有代码影响

## ADDED Requirements

### Requirement: 知识库构建系统
系统应提供从Excel数据构建向量知识库的功能。

#### Scenario: 成功构建知识库
- **WHEN** 系统读取Excel文件中的论文数据
- **THEN** 系统应将每篇论文的标题、关键词、摘要等信息转换为向量并存储到向量数据库中

#### Scenario: 数据预处理
- **WHEN** 系统处理论文数据
- **THEN** 系统应清洗数据、合并文本字段（标题+关键词+摘要），并生成文档元数据

### Requirement: 语义检索功能
系统应提供基于语义相似度的文献检索功能。

#### Scenario: 成功检索相关文献
- **WHEN** 用户输入论文主题
- **THEN** 系统应返回按相关性排序的Top 3篇文献

#### Scenario: 检索结果包含元数据
- **WHEN** 系统返回检索结果
- **THEN** 结果应包含文献标题、作者、年份、期刊、关键词、摘要等完整信息

### Requirement: 智能总结功能
系统应使用大语言模型对检索到的文献进行核心观点总结。

#### Scenario: 成功生成总结
- **WHEN** 系统获取到检索结果
- **THEN** 系统应调用大模型生成每篇文献的3个核心观点

#### Scenario: 总结质量要求
- **WHEN** 大模型生成总结
- **THEN** 每个观点应准确、简洁（≤100字），使用学术但易懂的语言，并按重要性排序

### Requirement: Web交互界面
系统应提供Web界面供用户交互。

#### Scenario: 用户输入主题
- **WHEN** 用户在Web界面输入论文主题并提交
- **THEN** 系统应显示加载状态并开始处理

#### Scenario: 结果展示
- **WHEN** 系统完成处理
- **THEN** 界面应结构化展示文献列表和核心观点总结

### Requirement: 防止幻觉行为
系统应确保大模型总结基于实际文献内容，避免生成虚假信息。

#### Scenario: 基于实际内容总结
- **WHEN** 大模型生成总结
- **THEN** 总结内容必须基于检索到的文献实际内容，不得编造或夸大

## 技术架构说明

### 核心技术栈选择

#### 1. LangChain框架
**选择理由**：
- LangChain是构建LLM应用的标准框架，提供了完整的RAG实现工具链
- 内置文档加载器、文本分割器、向量存储、检索器等组件，开发效率高
- 支持多种大模型API，便于切换和扩展
- 社区活跃，文档完善，问题容易解决

**其他选择**：
- LlamaIndex：专注于数据索引和检索，但在应用构建方面不如LangChain灵活
- 自研方案：开发周期长，稳定性难以保证

#### 2. 向量数据库：ChromaDB
**选择理由**：
- 轻量级，无需额外安装数据库服务，适合Demo项目
- 支持本地持久化存储
- 与LangChain集成良好
- 开源免费

**其他选择**：
- FAISS：Facebook开源的向量检索库，性能高但功能相对简单
- Pinecone：云服务，需要付费，适合生产环境
- Milvus：功能强大，但部署复杂，适合大规模应用

#### 3. Embedding模型：text-embedding-v3
**选择理由**：
- 阿里云通义千问提供的Embedding模型，与使用的大模型API一致
- 中文效果好，适合处理中文学术文献
- API调用简单，无需本地部署模型

**其他选择**：
- OpenAI text-embedding-ada-002：英文效果好，中文效果一般
- 本地模型（如m3e-base）：需要本地部署，资源消耗大

#### 4. 大语言模型：通义千问（Qwen）
**选择理由**：
- 用户已提供API密钥，无需额外申请
- 中文理解和生成能力强，适合学术文献总结
- 支持长上下文，可以处理较长的文献摘要
- 价格合理，适合学生使用

**其他选择**：
- OpenAI GPT系列：英文效果好，但中文效果不如国产模型
- 文心一言：中文效果好，但需要单独申请API
- ChatGLM：开源模型，可本地部署，但需要GPU资源

#### 5. Web框架：Streamlit
**选择理由**：
- 专为数据科学和AI应用设计，开发速度快
- 无需前端知识，纯Python开发
- 内置组件丰富，支持Markdown渲染
- 适合快速构建Demo

**其他选择**：
- Gradio：类似Streamlit，但自定义能力稍弱
- Flask/Django：功能强大，但开发周期长，需要前端知识
- FastAPI + Vue：前后端分离，适合生产环境，但开发复杂

#### 6. 数据处理：Pandas + OpenPyXL
**选择理由**：
- Pandas是Python数据分析的标准库，功能强大
- OpenPyXL支持Excel文件读写
- 与LangChain的文档加载器配合良好

**其他选择**：
- xlrd/xlwt：老牌Excel库，但功能不如OpenPyXL
- polars：性能更好，但生态不如Pandas成熟

### 数据流程

1. **知识库构建阶段**：
   - 读取Excel文件 → Pandas DataFrame
   - 数据清洗和预处理 → 合并文本字段
   - 文本分割 → LangChain TextSplitter
   - 向量化 → DashScope Embedding API
   - 存储 → ChromaDB向量数据库

2. **用户查询阶段**：
   - 用户输入主题 → Streamlit界面
   - 查询向量化 → DashScope Embedding API
   - 语义检索 → ChromaDB检索Top 3
   - 构建Prompt → LangChain PromptTemplate
   - 调用大模型 → DashScope Chat API
   - 结构化输出 → Streamlit展示

### 防止幻觉的策略

1. **严格的Prompt设计**：
   - 明确要求"根据提供的文献内容总结"
   - 禁止编造或推测信息
   - 要求观点必须有文献依据

2. **提供完整的上下文**：
   - 将检索到的文献完整信息传递给大模型
   - 包含标题、作者、摘要等所有可用信息

3. **输出验证**：
   - 检查输出格式是否符合要求
   - 可以添加引用来源标注
